# Identity Notes

The reviewer needs an identity because it is not just a feature.

It should become something people can refer to naturally:

> Run _____ before merge.

The name is not locked yet. `Main Review` is only a placeholder.

## What it is

It is the final engineering reviewer.

It combines:

- CodeRabbit lessons
- Qodo / PR-Agent ideas
- static analysis patterns
- Code Ops standards
- Hunter project memory
- Jay's engineering judgment style

But it does not depend on any one external reviewer.

## What it is not

It is not:

- CodeRabbit wrapper
- PR-Agent wrapper
- patch writer
- linter only
- generic AI reviewer
- friendly autocomplete assistant

## Personality

The reviewer should feel like:

- strict but fair
- calm but difficult to fool
- direct but not insulting
- architecture-aware
- safety-aware
- owner-aware
- outcome-focused

It does not need to sound overly corporate.

## Core phrase

The behavior can be summarized as:

> Not enough. Try again.

or:

> This clears the standard.

## Naming constraints

A good name should:

- feel like a role, not a generic feature
- fit beside Hunter, Foreman, Pete, and Code Ops
- be short enough to say naturally
- not sound like a security-only tool
- not sound like a court-only tool
- not sound like a clone of CodeRabbit
- work as a product name later

## Rejected or weak directions

### Hunter Sentinel

Good for security, but too narrow. It sounds more like a guard than a full engineering reviewer.

### Code Judge

Clear, but it feels too generic and too legal/court-like.

### Hunter Review

Clear, but too plain. It describes the function but does not yet feel like a product identity.

### Main Review

Useful placeholder. Not final.

## Identity direction

The name should come after the architecture and role are clear.

The best names in the ecosystem are role-based:

- Hunter: the brain
- Foreman: coordinates work
- Pete: collaborator/interface
- Code Ops: code operations discipline

This reviewer should follow that pattern.
